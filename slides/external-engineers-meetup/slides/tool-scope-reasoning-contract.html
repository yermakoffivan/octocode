Bilbo tools and MCPs share the Octocode protocol.

Every Bilbo tool — code, data, logs, docs — speaks the same envelope. Reduces the logical burden on the agent when choosing and calling tools: one shape, one mental model, every domain.

Why it works

The agent doesn’t relearn a calling convention per tool. Same envelope, same discipline — the model carries one cognitive posture across the whole investigation. Less context spent on tool semantics, more on reasoning.

The headline: every Bilbo tool and MCP we ship shares the Octocode protocol. Not by accident — by design. The point of the shared envelope is to reduce the logical burden on the agent. If each tool had its own calling convention, the model would spend context relearning shapes; here it carries one shape across every domain. Three layers to walk through. Context (input side): smart instructions, tool descriptions, and strict zod schemas tell the agent when and how to reach for the tool — that's how it picks the right tool without guessing. Execution: every call carries id, mainResearchGoal, researchGoal, and reasoning before any tool-specific args. That forces intent at the call site — the model declares why this tool now, before doing any I/O. Response: the result isn't a raw dump; it carries hints (what to do next) and pagination (bounded reads). The response steers the loop instead of flooding context. Finally — the research prompt itself is aware of common workflows: routing rules, cheapest-tool-first, discover → search → read. The agent doesn't reinvent the workflow each turn; the prompt encodes the well-trodden paths so the model can specialize on the actual investigation.
