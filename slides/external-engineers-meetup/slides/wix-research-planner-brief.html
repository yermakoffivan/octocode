The Planner is domain-agnostic — it doesn't investigate, it shapes. One turn of context engineering whose only job is to understand three things from the ask: intent (root_cause / trace_flow / research …), output (markdown / json / csv / code — the schema for the final artifact), and research requirements (goal, context, critical unknowns). Its purpose is to optimize the research pipeline: the researcher should never start cold, never guess identifiers, never lean on model memory for FQDNs or topology. To do that, the Planner runs three lookup lanes in parallel — memory (get_memory), docs (getFqdnData / getDocsData), Jira (getJiraContext → imageToText, only on a real key/URL) — and gates the whole thing on whether there's an anchor worth chasing. Output is one strict PlannerOutputEmissionSchema JSON, validated by zod with .strict() — five required, five optional, omission carries meaning. The researcher consumes it verbatim.
