tools never stagnate
We made a deliberate decision early: every tool the agent uses is built as an MCP — not a hand-rolled SDK call inside the agent process. Two reasons. First, MCP gives us a clean protocol (tool list, schemas, structured hints) that maps directly onto the agent’s reasoning loop. Second, and this is the bigger one: the same MCPs are usable by Wix engineers in their IDE or CLI for manual investigation. So we ended up with one shared surface and two consumers. Bilbo’s planner, researcher, and helpers call MCP tools through the agent loop. Engineers call the same MCPs by hand to triage incidents, validate findings, or just look something up. That dual usage created a continuous-improvement loop: when an engineer hits a missing hint, a vague schema, or a sharper output they want, they file it — and the next agent run gets it for free. Tools never stagnate, because humans are using them every day. The full set, all wired through one resolver: octocode for code, PRs and symbols across GitHub; service-topology for live runtime topology, pods, Kafka and gRPC reflection; grafana for dashboards, logs and deployment annotations; error-tracking for exception clusters; trino for SQL exploration over data warehouses; jira for ticket context including screenshots; api-docs for FQDN lookup and SDK packages; memory for session-scoped vector recall of prior investigations; users-data for experiments; personalization for placements. Helpers narrow this set per-call — the trino helper only sees trino, the grafana helper only sees grafana — so each subagent reasons inside a tight tool budget.
