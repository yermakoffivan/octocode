Why Bilbo? We run on many questions — code root cause, ownership, incident triage, rollouts, data lookups, topology, logs, tickets, docs, end-to-end flows — and they all reach for the same kinds of evidence: GitHub, service topology, API docs, Trino, Jira, experiments, placements, memory. Instead of N bespoke tools per team, Bilbo is one unified research service running one investigation pipeline. Three audiences call it: (1) Services — fire-and-forget async research over gRPC/Kafka; (2) Humans — the debugger UI and chat with follow-ups, sharing, forking; (3) Agents — MCP/HTTP for coding agents that need cited evidence, not guesses. Same planner, same researcher, same memory — different entry points. Next slide shows what the pipeline actually looks like.
