# Developer Review

Feedback from the agent to the **human who authored its operating instructions**:
root and `.octocode/AGENTS.md`, this SKILL.md, the system prompt, and the task brief.
Use it when the *instructions* caused time loss or a wrong turn.

Developer review closes the self-improvement loop. Agents identify ambiguous, wrong, over-constraining, or missing guidance, and the instruction author gets an evidence-backed fix list.

## Which Feedback Target

Reflection has three targets. Pick by *what would have to change to prevent the problem next time.*

| Flag | Fixes | Lands as | Read by |
| --- | --- | --- | --- |
| `--fix-repo` | the code / repo | `good`/`bad` refinement (coding queue) | the next coding agent |
| `--fix-harness` | the skill, tool, or hook machinery | `harness`-tagged memory | `reflect export-harness`; a human merges |
| `--fix-instructions` | the words that told you what to do | `instructions` refinement + `developer-review` memory | `reflect developer-review`, `.octocode/DEVELOPER_REVIEW.md` |

If you find yourself guessing a value the instructions never gave, following a rule that produced a worse result, or re-reading a brief three times to parse it — that is `--fix-instructions`, not `--fix-repo`.

## Record It

```bash
octocode-awareness reflect record --agent-id "$OCTOCODE_AGENT_ID" \
  --task "add lock retry" --outcome partial \
  --fix-instructions "AGENTS.md never states the default lock TTL; I guessed 300s and it expired mid-edit. Document the TTL and how to extend it." \
  --workspace "$PWD" --compact
```

One `--fix-instructions` call writes two records: a durable memory tagged `developer-review` / `instructions`,
and an `instructions` refinement with an open -> done lifecycle.
The refinement is hidden from the coding `refinement get` queue because it is addressed to the operator.
The default queue reports `instructions_count`.

## Consume It

- `octocode-awareness reflect developer-review --workspace "$PWD" --format markdown` — the human-facing digest, grouped Open / Resolved.
- `octocode-awareness query developer-review --workspace "$PWD"` — same rows as JSON for scripts.
- `.octocode/DEVELOPER_REVIEW.md` — regenerated by `repo inject`; indexed in the `.octocode/AGENTS.md` **Retro Files Map**. Regenerate it, never hand-edit.
- The `DeveloperReview` column on `query workboard` shows open feedback so agents see what the operator still owes an answer to.

Resolve an item once the instructions are updated: `octocode-awareness refinement set --refinement-id <id> --state done`.

## Write Feedback That Earns a Fix

- **Name the source and the cost.** "AGENTS.md §Locks" + "guessed TTL, lock expired mid-edit" beats "docs unclear."
- **Propose the fix.** Say what the instruction *should* say, not just that it was wrong.
- **Attach files** with `--fix-file` when the feedback is about a specific instruction file, so it shows in the file map.
- **One concern per call.** Separate items stay separately actionable and resolvable.

## Dogfood

When the repo you are editing IS this awareness tooling, this channel is how the tool improves from its own use.
Operate through awareness (`attend`, `lock`, `signal`).
When its own instructions fail, record `--fix-instructions` against them.
Strong guidance reduces repeat developer-review items.
