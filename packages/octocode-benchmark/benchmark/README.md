# Octocode benchmark suite

Benchmark source folders are split by measurement surface: offline engine
checks, offline CLI metadata checks, optional comparisons, live Octocode tool
questions, and cross-repo probes. Engine checks exercise the shipped
`octocode-engine` napi binary; CLI checks exercise built agent-facing help, tool
schemes, and OQL schema output. This suite lives in the private
`@octocodeai/octocode-benchmark` workspace package.

> **Full format support:** the canonical, machine-verified matrix of every
> supported extension (AST / signature / LSP / minify) is generated by
> `yarn support:gen` (`check-matrix.mjs --write`) **into the docs LSP guide** —
> [`docs/LSP_SERVER_LIFECYCLE.md` → Full format support matrix](https://github.com/bgauryy/octocode/blob/main/docs/LSP_SERVER_LIFECYCLE.md#full-format-support-matrix)
> (between the `support-matrix` markers). That is what the repo README points to.

## Run

```bash
yarn workspace @octocodeai/octocode-engine build:dev
node benchmark/run-all.mjs        # default benchmark/test checks
# or individually:
node benchmark/cli/check-cli-metadata.mjs
node benchmark/ast/check-ast.mjs
node benchmark/ast-grep/compare-upstream-scenarios.mjs --sync-repos --scenario gin-middleware-routing
node benchmark/lsp/check-lsp.mjs
node benchmark/minify/check-minify.mjs
node benchmark/repo/clone.mjs
node benchmark/repo/run.mjs
```

Equivalent yarn scripts: `yarn cli:check`, `yarn ast:check`, `yarn lsp:check`,
`yarn minify:check`, `yarn ast:compare:upstream`, `yarn benchmark`, and
`yarn test` (run-all). Engine package compatibility scripts delegate here. The
ast-grep comparison is optional and is not part of `yarn benchmark` because it
needs external CLIs and cloned upstream scenario repositories.

The default runner includes the offline CLI metadata gate for all command
descriptions/schemes, raw tool descriptions/schemes, agent context instructions,
and OQL schema. Live CLI/tool flow benchmarks remain manual question-catalog
checks rather than default runner steps, so network, npm, clone, and
auth-dependent checks do not make the benchmark flaky. See the
[split CLI/tool/OQL question catalog](https://github.com/bgauryy/octocode/blob/main/packages/octocode-benchmark/benchmark/octocode/README.md)
for GitHub, npm, clone-backed local proof, minify, OQL, parity, and workflow
checks. For durable smoke artifacts, run
`node packages/octocode-benchmark/benchmark/octocode/run-live-smoke.mjs` (or
`--skip-network` for local-only lanes), then
`node packages/octocode-benchmark/benchmark/octocode/validate-output-runs.mjs`.
Agents must write reported runs to
`packages/octocode-benchmark/output/<benchmark-name>-<YYYYMMDDTHHMMSSZ>/` and
follow the
[agent benchmark runbook](https://github.com/bgauryy/octocode/blob/main/packages/octocode-benchmark/recipes/agent-benchmark-runbook.md).
The machine-readable `summary.json` file must follow
[output-run.schema.json](https://github.com/bgauryy/octocode/blob/main/packages/octocode-benchmark/benchmark/output-run.schema.json).

## Real samples, not invented

The `ast/` and `lsp/` samples are real third-party source files fetched from
public repositories (TypeScript compiler, Rust core, Spring, LLVM, nvm,
Bootstrap, Home Assistant, …). Each `manifest.json` records the exact repo,
branch, path, byte size, and sha256 so a sample can be re-fetched or audited.
Files retain their upstream licenses and are used only to exercise the engine.

## Notes

- Tree-sitter is the **only** signature path; there is no regex/heuristic
  fallback (the former `src/signatures/heuristic.rs` was orphaned and deleted).
  Structural-only grammars (HTML/CSS/SCSS/Less/Scala/JSON/YAML/TOML) produce no
  signature outline by design.
- Scala has a grammar (structural search works) but **no configured language
  server**, so it appears in `ast/` but not `lsp/`.
- The LSP check verifies wiring + native (in-process) semantics only. Real
  server-backed features need the external servers installed; the check reports
  their availability but never fails on a missing server.
